# Sea Salt Vanilla tooling guide

[English](README.md) | [简体中文](README.zh-CN.md)

The scripts in this directory maintain the generated project data and public documentation for the pack. They are intended to be run from the repository root.

## Source-of-truth model

- `docs/config/` is the source for the public project matrix and classification intent.
- `packwiz` metadata is the installation fact used by checks.
- `cache/modrinth/` stores local Modrinth metadata cache files generated by `tools/refresh_modrinth_cache.py`; they are ignored by git and are not authored truth.
- `data/*.json` and `docs/*.md` are generated outputs, not sources of truth.

Do not hand-edit generated files unless you are debugging the generator itself.

## Recommended commands

Generate everything after editing `docs/config/` or packwiz metadata:

```bash
python tools/refresh_modrinth_cache.py
python tools/update_project_data.py generate
```

Check everything before committing:

```bash
python tools/update_project_data.py check
```

The `check` command verifies repository-internal consistency invariants without writing generated outputs or cache/report files. It does not regenerate `data/*.json` or `docs/*.md`. Default CI check does not read Modrinth cache and does not call the Modrinth API; regeneration and cache-backed diagnostics should be run explicitly outside the default CI check.

For the detailed check boundary, table, and examples, see [`CHECK_MODEL.zh-CN.md`](CHECK_MODEL.zh-CN.md).

## Generated data semantics

- `data/projects.json`: default projects declared by docs config matrix files.
- `data/optional.json`: optional projects and alternatives declared by docs config.
- `data/dependencies.json`: dependency-only projects maintained by the explicit dependency-generation step. Default CI check treats it as committed generated data for classification coverage instead of re-deriving dependencies from Modrinth cache.

The checker expects these sets to stay disjoint. It also verifies:

- `projects.json + optional.json = docs/config project refs`
- `projects.json`, `optional.json`, and `dependencies.json` are mutually disjoint
- `projects.json + dependencies.json = packwiz default installed project refs`, meaning installed-project classification coverage rather than CI-time dependency re-derivation
- no duplicate project refs exist in the target-version docs config

## Script reference

### `update_project_data.py`

Main entry point for day-to-day work.

```bash
python tools/update_project_data.py generate
python tools/update_project_data.py check
```

Single-step commands are also available:

```bash
python tools/update_project_data.py projects
python tools/update_project_data.py dependencies
python tools/update_project_data.py docs
```

Single-step commands generate their corresponding output. The normal `generate` command runs `projects`, `dependencies`, and `docs`. To see whether generation changed anything, run the generator and inspect `git diff`.

### `refresh_modrinth_cache.py`

Refreshes the Modrinth metadata snapshot used by generation and optional diagnostics.

```bash
python tools/refresh_modrinth_cache.py
```

This script is the explicit network-enabled entry point for project-data tooling. It scans packwiz metadata and docs config refs, then refreshes:

- `cache/modrinth/modrinth-version-dependencies.json`
- `cache/modrinth/modrinth-projects.json`
- `cache/modrinth/manifest.json`

If Modrinth queries fail, the script exits non-zero and leaves the existing cache files unchanged. Transient API errors are not written into the local cache; refresh successfully before running cache-backed generation or diagnostics.

Useful modes:

```bash
# Fill only missing snapshot data.
python tools/refresh_modrinth_cache.py

# Re-fetch every required Modrinth version/project entry.
python tools/refresh_modrinth_cache.py --force

# Rebuild only project metadata, useful after changing project-cache fields.
python tools/refresh_modrinth_cache.py --only-projects --force --verbose

# Inspect scope without network or writes.
python tools/refresh_modrinth_cache.py --dry-run --verbose
```

Cache files are generated artifacts used for generating `data/dependencies.json` and enriching project metadata. They are local, gitignored, and disposable. Default CI check does not depend on them. Do not hand-edit them; refresh them with the script.

`modrinth-version-dependencies.json` is keyed by Modrinth version id and stores the locked version fields used by checks: project id, version number, loaders, files, and version dependencies. `modrinth-projects.json` uses a normalized structure to avoid duplicating project objects:

```json
{
  "schema_version": 2,
  "projects": {
    "5faXoLqX": { "id": "5faXoLqX", "slug": "iceberg", "title": "Iceberg" }
  },
  "aliases": {
    "iceberg": "5faXoLqX"
  },
  "errors": {}
}
```

Manifest counters have precise meanings:

- `installed_projects`: packwiz project files scanned from supported project folders.
- `version_refs`: unique installed Modrinth version ids found in packwiz metadata.
- `version_cache_entries`: entries currently stored in `modrinth-version-dependencies.json`.
- `project_refs`: unique project lookup refs required by docs config, optional entries, matrix alternatives, installed locks, and required dependency closure.
- `project_refs_to_fetch`: project refs that were missing, errored, or force-refreshed for this run.
- `project_cache_projects`: unique project metadata objects stored under `projects`.
- `project_cache_aliases`: alias lookup keys, such as slugs, mapped to project ids.

### `generate_project_registry.py`

Generates:

- `data/projects.json`
- `data/optional.json`

Inputs:

- `docs/config/**/matrix/*.json`
- `docs/config/**/optional.json`
- local Modrinth project snapshot for metadata enrichment

This script does not require live network access.

### `generate_project_dependencies.py`

Generates:

- `data/dependencies.json`

Inputs:

- packwiz metadata under `mods/`, `resourcepacks/`, `shaderpacks/`, `datapacks/`, and other supported project folders
- docs config default roots, optional groups, and matrix alternatives
- Modrinth version dependency snapshot

Generation never calls the network or writes the cache. If required snapshot data is missing, it prompts you to run `python tools/refresh_modrinth_cache.py` first. Default CI check does not run cache-backed dependency regeneration.

### `generate_project_docs.py`

Generates bilingual public docs:

- `docs/mods.md`
- `docs/mods.zh-CN.md`
- `docs/resourcepacks.md`
- `docs/resourcepacks.zh-CN.md`
- `docs/shaderpacks.md`
- `docs/shaderpacks.zh-CN.md`
- `docs/datapacks.md`
- `docs/datapacks.zh-CN.md`

Inputs:

- `docs/config/**/meta.json`
- `docs/config/**/matrix/*.json`
- `docs/config/**/optional.json`
- generated project catalogs

### `check_project_data.py`

Checks generated data, docs config, packwiz metadata, and project uniqueness. This script only checks; it does not write files, read Modrinth cache, or access the Modrinth API.

### `check_generated_docs.py`

Checks generated Markdown docs without writing files. It renders expected docs in memory and compares them with committed `docs/*.md` files.

`python tools/update_project_data.py check` runs both check scripts: repository consistency plus generated-doc freshness. Both check scripts are read-only.

Important summary fields:

- `unexplained`: installed packwiz projects not explained by docs or dependency data.
- `cache_errors`: unresolved Modrinth cache/API errors.
- `generated_data_invariants`: broken set-equation or disjointness rules.
- `missing_defaults`: docs default projects missing from packwiz.
- `unexpected_installed`: optional or alternative projects installed in the default pack.
- `unknown_refs`: docs refs missing from generated catalogs.
- `duplicate_refs`: the same project appears more than once in target-version docs config.

### Shared helper modules

- `project_data_common.py`: shared filesystem, packwiz, Modrinth, dependency, and markdown helpers.
- `project_data_identity.py`: shared project identity and reference-resolution helpers.
- `project_data_invariants.py`: generated-data set invariant checks.

These helper modules are not meant to be run directly.

## Suggested workflow

1. Edit `docs/config/` first.
2. If you add or update Modrinth projects/versions, run `python tools/refresh_modrinth_cache.py`.
3. Run `python tools/update_project_data.py generate`.
4. If the checker reports missing or unexpected packwiz projects, update packwiz metadata accordingly.
5. Refresh the Modrinth snapshot again if needed, then rerun `python tools/update_project_data.py generate`.
6. Run `python tools/update_project_data.py check` before committing.
